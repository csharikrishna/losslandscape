# The Loss Landscape — an interactive expedition

A scroll-driven, 3D educational website teaching the geometry of neural
network optimization: basins, saddles, ridges, cliffs, and why some minima
generalize and others don't. Built with Next.js, React Three Fiber, and
Framer Motion. Every terrain, chart, and icon is generated by code — there
are no stock images or generated raster illustrations anywhere in the
project.

## Quick start

```bash
npm install
npm run dev
```

Then open **http://localhost:3000**.

```bash
npm run build   # production build
npm run start   # serve the production build
```

Requires Node.js ≥ 18.17.

> **A note on how this was built:** this project was written in a sandboxed
> environment with no network access, so dependencies could not be installed
> or the build tested end-to-end before delivery. Every package version in
> `package.json` is a real, released version that's mutually compatible as
> of mid-2024, and the code follows standard, well-documented patterns for
> each library (see "Implementation notes" below) — but budget a few minutes
> after your first `npm install` to confirm the build, and see
> **Troubleshooting** if anything needs a small nudge.

## What's actually happening on the page

One `<canvas>` (React Three Fiber) is fixed behind the entire page and never
unmounts. Its terrain is a real heightmap — not an image — generated once
from a hand-rolled noise function plus a set of deliberately placed
"features" (a sharp well, a saddle, a plateau, a cliff, a ridge, a ravine,
two minima) defined in `lib/terrain-data.ts`. A glowing line is drawn through
those same features and used two ways: as a lit path across the terrain, and
as the literal scroll-progress indicator — the dot marking "you are here"
advances along it as you scroll. A camera rig flies a spline through the
same landscape, driven by the same scroll value, so the whole page reads as
one continuous flight rather than a series of separate embedded 3D widgets.

Every content section is a normal DOM element that scrolls **on top of**
that fixed canvas, in a translucent glass panel, so the terrain is always
at least partially visible behind the reading content.

## Project structure

```
app/
  layout.tsx          Root layout: fonts (next/font/google), metadata
  page.tsx             Assembles the canvas stage + every section, in order
  globals.css          Design tokens, glass panels, grain overlay, scrollbar
  icon.svg             Favicon (Next.js auto-detects this)

components/
  canvas/              Everything inside the <Canvas>
    CanvasStage.tsx      The fixed, full-viewport wrapper (client-only)
    Scene.tsx            Lights, stars, fog color, bloom post-processing
    Terrain.tsx           Heightmap geometry + the elevation/contour shader
    TrajectoryPath.tsx    The glowing path + "you are here" marker
    CameraRig.tsx          Scroll → camera spline
  sections/            One component per page section (Hero, SaddlePoint, …)
  diagrams/            The bespoke SVG/canvas illustrations (all hand-authored)
  ui/                  Small shared primitives (Panel, Reveal, MathTag, …)

lib/
  terrain-data.ts      Single source of truth: terrain features + journey
                       waypoints, shared by the mesh, camera, and trajectory
  trajectory.ts        Builds the Catmull-Rom curve through the waypoints
  noise.ts             Dependency-free 2D value-noise / FBM
  shaders.ts           GLSL source (as template strings) for the terrain
                       and trajectory materials
  scroll-context.tsx   Document scroll progress, exposed via React context
  motion.ts            prefers-reduced-motion hook
  utils.ts             clamp / lerp / remap helpers

content/
  loss-landscape.md    The source educational content (also the PDF source)

public/pdf/
  loss-landscape.pdf   Generated from content/loss-landscape.md
```

## Customizing the terrain

Everything about the journey — the narrative order, the camera framing, and
the terrain's actual shape — is defined in **one file**:
`lib/terrain-data.ts`.

- `JOURNEY` is the ordered list of stops. Each has an `(x, z)` position and
  camera framing (`camHeight`, `camBack`, `camSide`). Add a stop here (and a
  matching `<section id="...">` in `app/page.tsx`) to add a new leg of the
  journey — the camera rig and trajectory line pick it up automatically.
- `FEATURES` is the list of terrain bumps/wells/saddles/etc. that actually
  shape the mesh. Each `TerrainFeature` has a `kind` (`well`, `saddle`,
  `plateau`, `cliff`, `ridge`) and a handful of numeric params (position,
  radius, amplitude). `heightAt(x, z)` in the same file sums all of them
  plus a base fractal-noise layer — read that function for exactly how each
  kind is computed.
- `CAMERA_SCROLL_END` in `lib/trajectory.ts` (default `0.8`) controls what
  fraction of total page scroll the flythrough takes to complete; the
  remaining fraction holds the camera at the final "overlook" waypoint while
  the closing sections (concept grid, takeaways, PDF, footer) scroll past.

## Regenerating the PDF

The PDF is generated from `content/loss-landscape.md`. If you edit that file
and want to regenerate `public/pdf/loss-landscape.pdf`, the simplest path on
most systems with `pandoc` and `wkhtmltopdf` installed:

```bash
pandoc content/loss-landscape.md --from markdown --to html5 -o /tmp/content.html
# wrap /tmp/content.html with your own <style> block (see the design notes
# for the token values used on the site), then:
wkhtmltopdf --enable-local-file-access /tmp/full.html public/pdf/loss-landscape.pdf
```

Any Markdown → PDF pipeline works fine — Pandoc direct to PDF, a headless
Chrome print-to-PDF, `md-to-pdf`, etc. There's nothing else in the repo that
depends on how the PDF was produced.

## Implementation notes (what to check first if something's off)

- **3D materials**: the terrain and trajectory use `THREE.ShaderMaterial`
  passed directly via the `material` prop of `<mesh>`, and `<shaderMaterial>`
  as a JSX intrinsic (React Three Fiber auto-registers every `THREE.*`
  export as a lowercase-first JSX tag — no manual `extend()` needed for
  either).
- **Camera/trajectory sync**: both `CameraRig` and `TrajectoryPath` read the
  same smoothed scroll value from `lib/scroll-context.tsx` and the same
  `toCameraProgress()` remap from `lib/trajectory.ts`. If you change section
  heights and the camera timing feels off, that remap (and `CAMERA_SCROLL_END`)
  is the first place to look.
- **Fonts**: loaded via `next/font/google` (Space Grotesk, IBM Plex Sans,
  IBM Plex Mono) in `app/layout.tsx` — this downloads and self-hosts the
  fonts at build time, so the deployed site never calls out to Google Fonts
  at runtime.
- **No SSR for the canvas**: `CanvasStage` is imported with
  `next/dynamic(..., { ssr: false })` in `app/page.tsx` since WebGL needs a
  browser.

## Troubleshooting

- **"Module not found" on first install** — double check your Node version
  (`node -v`, need ≥ 18.17) and delete `node_modules` + lockfile before
  retrying if you switched Node versions mid-way.
- **Terrain is black / shader errors in the console** — almost always a
  uniform name mismatch between `lib/shaders.ts` and where a material is
  constructed (`Terrain.tsx` / `TrajectoryPath.tsx`). Every uniform the
  fragment shader references must have a matching entry in that component's
  `uniforms` object.
- **Bloom looks too strong/weak or tanks frame rate on a low-end GPU** — tune
  or remove the `<Bloom>` element in `components/canvas/Scene.tsx`;
  `mipmapBlur` is the most expensive setting to drop first.
- **Fonts look off in dev** — first `next dev` run needs a moment to fetch
  and cache the Google Fonts files; refresh once after the terminal shows
  "Compiled".
